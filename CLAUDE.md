# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A static, frontend-only website (no build step, no dependencies) for 焼津祭 (Yaizu Festival) showing the schedule/timetable and a map reference. Open `index.html` directly in a browser to preview.

## Architecture

- `data.js` — all content (`roles`, `schedule`, `mapPins`, etc.). This is the **only file meant to be edited** for routine updates; look for `TODO:` markers.
  - `roles` / `schedule` mirror an Excel source file managed in Dropbox — that file is never stored locally; when updating, refer to the Dropbox file directly (the share link is kept privately by the site owner, not documented here).
  - `mapPins.points` is an array of `{ label, lat, lng, mapUrl }`. Rendered as a list in the map tab; `mapUrl` (optional) adds a "view in Google Maps" link. There is currently no interactive map.
  - `mapPins.routeDiagram` renders a locally stored simplified route diagram (`mikoshi-route.svg`, blue=往路/green=帰路) above the official reference image in the map tab, plus a "簡略図のPDFを開く" link to `mikoshi-route.pdf`. The SVG is generated from `mikoshi-route.dio` (a draw.io source file) by `gen_route_svg.py`; the PDF is generated from the SVG by `gen_route_pdf.sh`.
  - `mapPins.officialRouteImage` hotlinks a reference image (portable shrine route map) directly from the 焼津神社 official site — no local copy is stored, and the URL changes year to year, so verify it against the official site when updating.
  - `mapPins.excelRouteLine` is leftover data for a previous Leaflet-based interactive map and is currently unused.
  - `mikoshi-route-reference.jpg` is a downloaded copy of the official route map kept for provenance/comparison — untracked, not referenced by the site.
- `app.js` — pure rendering of `data.js` content into the DOM; no interactive map logic remains.
- `index.html` / `style.css` — structure and styling; not expected to need changes for routine content updates.
- `gate.js` — client-side password gate shown before `data.js`/`app.js` are loaded (deters casual visitors only; the repo itself must be made private for real access control, since `data.js` remains directly fetchable from a public repo regardless of this gate).
- `print.html` / `gen_schedule_pdf.sh` — generate the printable `schedule.pdf` from `data.js`'s `schedule` via headless Chrome. `print.html` measures actual rendered row heights to auto-paginate onto A4-landscape pages, splitting a day across pages with a `（n/total）` suffix in the heading when it doesn't fit on one page.
- `mikoshi-route.dio` / `gen_route_svg.py` — `gen_route_svg.py` parses the draw.io XML (`mxCell` vertices/edges: box position, fill/stroke color, and text; edge source/waypoints/target) with Python's stdlib only (no draw.io app, Node, or network access needed) and renders it straight to `mikoshi-route.svg`.
- `mikoshi-route.md` — human-readable list of the mikoshi route under two headings, `# 往路` (outbound) and `# 帰路` (return), each stop with one or more Google Maps share links (`maps.app.goo.gl/...`). The first link under each named stop is the source of truth for that stop's entry in `data.js`'s `mapPins.points` (往路) / `mapPins.pointsReturn` (帰路) — `{ label, lat, lng, mapUrl }`, in order; bare links with no stop name above them are reference-only and not reflected there. Like `mikoshi-route.dio`, this is a live file the owner may edit directly between (or within) sessions.
  - `mapPins.routeMapOutbound` / `routeMapReturn` (`mikoshi-route-outbound.jpg` / `mikoshi-route-return.jpg`) are locally stored images, one per leg, plotting the *full* `mikoshi-route.md` coordinates for that leg — including the bare/unnamed links that `points`/`pointsReturn` skip — as a straight-line overlay with numbered markers on an OpenStreetMap basemap (no road-snapping, unlike a Google Maps directions link). The numbers match the position of each named stop in `points`/`pointsReturn`'s rendered `<ol>`. Shown inline (zoomable, like the other route images) above each leg's point list, each with its own "往路のPDFを開く"/"帰路のPDFを開く" link (`mapPins.routeMapOutboundPdf`/`routeMapReturnPdf`) to a same-named `.pdf`.
    - Regenerate with `./gen_route_maps.py` (reads `route_resolved.json`/`points_out.json`/`points_ret.json`, writes both JPGs directly). Pillow isn't a system Python package, so it needs a throwaway venv (`python3 -m venv ...; .../bin/pip install pillow`) — run the script with that venv's Python, not the system one.
    - `route_resolved.json` (full per-leg polyline, named + bare points) and `points_out.json`/`points_ret.json` (named stops only, in display order) are checked-in caches mirroring `mikoshi-route.md` (all links, resolved to lat/lng) and `data.js`'s `points`/`pointsReturn` respectively — regenerate them (re-resolve any new/changed `maps.app.goo.gl` links per the `mikoshi-route.md` sync rule below, re-export `points`/`pointsReturn` from `data.js`) before running `gen_route_maps.py` whenever either source file changes, or the images will silently drift out of sync with the rest of the site.
    - `points_out.json`/`points_ret.json` must have the exact same entries, in the exact same order, as `data.js`'s `points`/`pointsReturn` — a mismatch doesn't error, it just silently numbers the map markers one off from the point list below them (has happened: `points_ret.json` was missing an entry `pointsReturn` had). After editing either JSON cache by hand, diff its labels against `data.js` before regenerating images.
    - Basemap: fetch raw OSM tiles (`https://tile.openstreetmap.org/{z}/{x}/{y}.png`) covering the leg's lat/lng bounding box (+ a small pad for marker offsets, see below) and stitch them into one image with Pillow — no API key needed. Crop tight to that bbox (not a fixed/generous area) so the route fills the frame.
    - Markers: numbered circles in stop order, drawn on top of the plotted line. When two stops share the same or near-identical coordinates (has happened — e.g. two labels reusing the same `maps.app.goo.gl` link), their circles will land on top of each other; nudge one a fixed pixel offset away so both are readable. Don't also draw a leader line/dot back to the shared true point — when the shared coordinate is legitimate (two named things at one real spot, not a data error), that connector just reads as a stray unexplained line between two unrelated-looking markers.
    - Labels: text is just the stop name (the number lives only on the marker circle). Since collision-avoidance placement can land a label some distance from its own marker, draw a thin grey leader line from each marker to the center of its own label box so the pairing stays unambiguous — draw the leader lines before the label boxes/marker circles so those opaque shapes cover the line's two ends cleanly.
    - Labels need real collision avoidance, not a fixed offset — try candidate positions around the marker (multiple compass directions/radii), and for each candidate reject it if it overlaps the image edge, another marker (any marker, not just its own), an already-placed label, or sits too close to the route line; pick whichever remaining candidate clears the line by the widest margin.
    - For a stop whose name ends in "〜通り" (it names the street travelled just after that point, not the point itself), bias its candidate search toward the route's actual heading leaving that point (computed from the full polyline, not just the named stops) instead of searching all directions equally — it should read as labelling the road leaving the marker in the direction of travel, falling back to nearby angles only if the preferred direction collides with something.
    - Route-line rendering resolution/thresholds are zoom-level-dependent (tile pixels don't have a fixed real-world size) — if the OSM zoom level used to build the basemap changes, the near-parallel-segment-detection distance threshold (see below) must be rescaled proportionally or it'll stop firing (or fire on the wrong things).
    - Near-parallel, near-overlapping non-adjacent segments (the recorded path runs a short loop and comes back close to itself) need visual separation, but do it by shifting a run of *existing* vertices sideways as a unit (a short diagonal transition in, a clean parallel run, a short diagonal transition back out) — not by bulging a single new point at the segment's midpoint, which reads as an unnatural sharp kink/spike rather than a clean offset.
    - Keep the parallel-segment detection threshold tight (`sep=22, close_thresh=34, angle_thresh_deg=25` in `build_display_path`, tuned for `ZOOM=18`) — only separate segments that would actually render as touching/coincident. It's fine for the route line to run close to or cross the *basemap* (roads, buildings); only route-vs-route overlap is a problem. A wider threshold was tried and reverted — it also moved vertices for segments that were merely near each other, not truly overlapping.
    - Direction arrows must be computed from the exact same point list that gets drawn as the line (after any parallel-segment separation, not before) — computing them from a different/earlier version of the path is what caused a past bug where arrows floated off to the side of the line and pointed the wrong way.
    - After regenerating either JPG, also regenerate its PDF: `mikoshi-route-outbound.pdf` / `mikoshi-route-return.pdf` are made via `route-map-print.html` (a generic print page taking the image path as a `?src=` query param and sizing `@page` to its native pixel dimensions), the same headless-Chrome print-to-pdf recipe as `route-print.html`/`gen_route_pdf.sh` — e.g. `--headless=new --disable-gpu --print-to-pdf=mikoshi-route-outbound.pdf --no-pdf-header-footer "http://localhost:PORT/route-map-print.html?src=mikoshi-route-outbound.jpg"` against a local server serving this directory.
- `route-print.html` / `gen_route_pdf.sh` — generate `mikoshi-route.pdf` from `mikoshi-route.svg` via headless Chrome. `route-print.html` sizes `@page` to the SVG's own pixel dimensions (read via the loaded `<img>`'s `naturalWidth`/`naturalHeight`) so the PDF is one page at the diagram's native aspect ratio, not cropped/scaled to A4.

## Rules

- **After editing `data.js`'s `schedule`, regenerate the PDF**: run `./gen_schedule_pdf.sh` from this directory to keep `schedule.pdf` in sync. It has drifted out of sync before (edits to `data.js` without regenerating the PDF), so don't skip this — including when `schedule` changes as a side effect of other work, not just when explicitly asked to edit it. When in doubt, check `git diff data.js` for `schedule` changes before wrapping up.
- **After editing `mikoshi-route.dio`, regenerate the SVG**: run `./gen_route_svg.py` from this directory to keep `mikoshi-route.svg` in sync with the diagram source.
- **After regenerating `mikoshi-route.svg`, regenerate the PDF too**: run `./gen_route_pdf.sh` from this directory to keep `mikoshi-route.pdf` in sync.
- **`mikoshi-route.dio` is a live file the owner edits directly in the draw.io desktop app between Claude sessions** (and sometimes between messages in the same session), not just through requests made here. Before doing route-diagram work — and especially when asked to "reflect the dio changes" — run `git diff mikoshi-route.dio` first rather than assuming it's unchanged; regenerate the SVG whenever it has changed, even if no one explicitly flagged an edit.
- **After editing `mikoshi-route.md`, sync `data.js`'s `mapPins.points`/`pointsReturn`**: for each named stop under `# 往路`/`# 帰路`, the label and its first `mapUrl` should match a `{ label, lat, lng, mapUrl }` entry in the matching array (`points` for 往路, `pointsReturn` for 帰路), in the same order (drop stops entirely if removed from the .md; add new ones with `lat`/`lng` resolved by following the `maps.app.goo.gl` redirect — `curl -sIL <url>` and read the `lat,lng` out of the final `location:` header). `mikoshi-route.md` is also a file the owner may edit directly between/within sessions — run `git diff mikoshi-route.md` before wrapping up any route-related work, the same way `mikoshi-route.dio` is checked, and resync even if no one explicitly flagged an edit. If the route itself changed (not just a stop's label), `mikoshi-route-outbound.jpg` / `mikoshi-route-return.jpg` should be regenerated from every link in the section (named or bare) too, since it's a separate manual step (see above).

## Notes

- No test suite, linter, or build/package manifest — intentionally dependency-free static HTML/CSS/JS.
- `焼津祭_2026.xlsx` and `.claude/` are gitignored.
- The current `data.js` schedule intentionally includes personal names/home names at the owner's request — if the visibility scope needs to change, edit that file directly.
- For local previewing without the password gate, create an `index_preview.html` alongside `index.html` — a copy that drops `#gate-overlay` and loads `data.js`/`app.js` directly instead of through `gate.js`. Serve the directory with `python3 -m http.server` and open that file. It's a throwaway dev aid, not committed.
